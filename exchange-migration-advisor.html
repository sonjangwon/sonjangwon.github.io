Exchange Migration Advisor

온프레미스 Exchange 환경 데이터를 입력받아 Microsoft 365 마이그레이션 가능 여부를 자동으로 평가합니다.
20개 결정적 룰 + 9개 자가진단 + 2개 전략 평가로 블로커를 사전 식별하고, Azure OpenAI가 환경 컨텍스트 기반 조치 가이드를 자동 생성합니다.

Problem

만들게 된 배경

엔터프라이즈 Exchange 마이그레이션의 비용은 문제 발견 시점에 비례합니다. cutover 직전에 발견된 블로커는 일정 지연뿐 아니라 이미 진행된 사용자 마이그레이션의 롤백 검토를 강제할 수 있습니다.

01 — Schedule Risk
2–3weeks
평균 일정 지연
용량 임계값 초과 사서함, 보존 정책 미정의 등 사전 작업이 필요한 항목들이 프로젝트 일정 지연을 초래합니다.
02 — Operational Risk
Servicedisruption
전사 서비스 중단 가능성
Public Folder 의존성 미식별, 인증서 만료, Hybrid 인증 실패 시 전사 메일 접근이 단절될 수 있습니다.
03 — Cost Risk
Manualconsulting
컨설턴트 의존
PowerShell cmdlet 수십 개 실행 후 결과 해석 — 컨설턴트 수준의 경험에 의존하는 사전 진단 단계입니다.

마이그레이션 프로젝트에서 가장 흔히 마주치는 패턴은 명확합니다.
첫째, 환경 파악에 며칠에서 몇 주가 소요됩니다 — PowerShell 명령어를 수십 개 실행하고, 그 결과를 종합 해석할 수 있는 사람이 별도로 필요합니다.

둘째, 진짜 블로커는 cutover 직전에 발견됩니다 — 1.5TB 초과 사서함, Public Folder 의존성, 만료된 인증서, OAuth 미구성 등. 어느 것도 마이그레이션 당일에 알아서는 안 되는 항목들입니다.

셋째, Microsoft 공식 문서는 양이 방대하고 고객사마다 환경 구성 · 운영 정책 · 보안 요건이 달라서 엔지니어가 명시적으로 마이그레이션 영향도에 대한 판단을 내리기 어렵습니다. Exchange Migration Advisor는 그 과정에 도움을 줍니다.

Approach

결정적 룰 + 생성형 AI 분업

마이그레이션 진단은 두 개의 서로 다른 작업으로 나뉩니다 — 결정적 사실 판단과 자연어 가이드 생성.
Exchange Migration Advisor는 이 둘을 분리해 다른 엔진이 처리하도록 설계했습니다.

Rule Engine Deterministic
  • 마이그레이션 블로커 식별
  • Severity 산출 — Blocker / Must Fix / Recommended
  • 6개 영역 분류 (organization, mailboxes, mailflow 등)
  • 전략 평가 (Cutover ≤150 / Hybrid >150)
Azure OpenAI Generative
  • 이슈별 자연어 조치 가이드 생성
  • 환경 컨텍스트 기반 PowerShell 명령 제안
  • Microsoft Learn 문서 검색 후 인용 (RAG)
  • 한국어 시스템 프롬프트로 톤·구조 일관성 유지
Why split the workload
LLM 단독으로는 결정적 사실을 신뢰할 수 없습니다 — 그럴듯하지만 잘못된 설명을 생성할 수 있어 마이그레이션 가능 여부 같은 정답이 정해진 판단에는 부적합합니다. Exchange Migration Advisor는 이런 결정적 영역을 명시적 룰로 처리해 정확한 기술적 근거에 기반한 일관된 답변을 보장하고, LLM은 그 결과를 사용자에게 자연스럽게 전달하는 역할에만 집중하도록 분리했습니다.
Data Collection

온프레미스에서 무엇을 가져오나

Exchange Migration Advisor의 입력은 PowerShell 수집기가 생성하는 표준 JSON입니다.
4개 모듈이 Exchange 환경의 핵심 정보를 영역별로 추출하고, schema_version: "1.0.0"으로 버전 관리되는 계약 형식으로 출력합니다.

01
조직 단위 정보

메일 흐름 구성

Get-ExchangeCertificate Get-AuthServer Get-FederationTrust
Self Assessment

PowerShell이 답할 수 없는 영역

모든 정보를 PowerShell로 수집할 수 있는 것은 아닙니다. Entra Connect는 Exchange가 아닌 별도 서버에서 동작하고, 네트워크 도달성은 클라이언트 위치와 방화벽에 따라 달라집니다. Exchange Migration Advisor는 이런 영역을 추측하지 않고 사용자에게 직접 물어봅니다.

Directory Sync 5 questions
  • DS-001Entra Connect 설치 및 동작 상태
  • DS-002마지막 동기화 시간과 주기
  • DS-003Staging Mode 여부
  • DS-004온프렘 ↔ Entra ID 사용자 수 일치
  • DS-005최근 동기화 오류 발생 여부
Network 4 questions
  • NET-001outlook.office365.com:443 도달성
  • NET-002EWS External URL 외부 도달 가능
  • NET-003Microsoft 365 IP/URL 허용 목록 최신화
  • NET-004SMTP Outbound (*.mail.protection.outlook.com:25)
Design Principle — Honest about Uncertainty
사용자가 답하지 않은 항목은 추측하지 않고 Unverified(미검증)로 분류되어 결과 화면에 명시적으로 표시됩니다. "확실하지 않은 것을 확실한 것처럼 답하지 않는다"는 진단 도구의 출력 신뢰성에서 가장 중요한 원칙이라고 판단했습니다.

AI Layer

Rule Engine이 식별한 이슈는 운영자에게 그대로 보여주기엔 너무 기술적입니다. Azure OpenAI의 "On Your Data" 기능을 통해 RAG 파이프라인을 통합했습니다.

Decisions

핵심 의사결정과 트레이드오프

도구를 만드는 과정에서 마주친 주요 분기점들입니다.각 결정은 더 좋은 다른 선택을 배제한 것이 아니라, Exchange Migration Advisor의 운영 환경과 도메인 특성에 맞게 의도적으로 선택한 결과입니다.

D-01
RAG vs Fine-tuning
엔터프라이즈 Exchange 환경은 고객사마다 구성과 운영 패턴이 다양해서, 단일 모델에 모든 사례를 학습시키기보다 도메인 문서를 인덱스로 분리하고 환경 컨텍스트에 맞춰 검색·인용하는 방식이 더 적합합니다. RAG는 검색된 문서를 답변과 함께 인용해 운영자가 출처를 직접 확인할 수 있고, 도메인 데이터를 모델과 분리해 인덱스만 갱신하면 즉시 반영됩니다. 답변 정확성과 운영 투명성이 중요한 엔터프라이즈 환경에서 더 안전한 접근입니다.
D-02
Rule + LLM Hybrid vs LLM-only
마이그레이션 블로커는 결정적 사실이 많습니다. LLM은 이런 영역에서도 확률적으로 답하기 때문에 환각이 발생할 수 있습니다. 결정적 규칙으로 사실 판단을 처리하고, LLM은 자연어 설명·환경 맥락 해석에만 집중하도록 책임을 분리한 결과 답변 신뢰도가 높아졌습니다.
D-03
Azure OpenAI vs OpenAI direct
엔터프라이즈 Exchange 환경 데이터는 사서함 정보·라우팅 구성 등 민감 데이터를 포함합니다. Azure OpenAI는 데이터 거주성(region 단위 배포), 프라이빗 엔드포인트(Private Link), Azure RBAC, 감사 로그 등 엔터프라이즈 거버넌스 요건을 충족하고, Entra ID 기반 인증으로 기존 M365 환경과 동일한 ID 체계를 사용할 수 있습니다.
D-04
Keyword Search vs Vector / Hybrid
Exchange 도메인은 정확한 cmdlet 이름(Set-CalendarProcessing), 오류 코드(500011), 제품 버전(CU23) 매칭이 검색 정확도에 결정적입니다. Vector 검색은 의미적 유사도에 강하지만 정확한 명칭 매칭이 필요한 도메인에서는 노이즈를 만들 수 있어, 키워드 검색을 채택했습니다.
